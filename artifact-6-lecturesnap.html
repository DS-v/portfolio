01 — The Problem

Why this domain is interesting — and where it's hard

Most AI products are judged by what they can do. The ones people rely on are judged by how they fail — and the worst failures are the silent ones, where the output looks complete and sourced but isn't, and the user can't tell until it's too late to matter.

LectureSnap turns lecture videos into exam-ready study notes for STEM students, a domain where a confidently wrong formula is worse than a missing one. I built the working product — Chrome extension, web app, and a five-layer processing pipeline — but the core of this artifact is what sits underneath it: 106 failure modes, grouped into 12 families and graded by how much trust each destroys, and the gap between the evaluation framework I designed and the minimal slice I actually wired and ran.

The success metric stays blunt: can a student pass the exam from the PDF alone, without rewatching the lecture?

02 — What I Built

Three surfaces, one five-layer pipeline

LectureSnap has three surfaces: a Chrome extension as the entry point (it scrapes captions and relays the user's session), a web dashboard for auth, credits, and PDF download, and an async pipeline worker doing the processing in five layers.

A 50-minute lecture becomes a 10–16 page PDF in 5–7 minutes for roughly $0.05–0.15 in model cost. Output is concept cards — definition, intuition, worked example, common pitfalls, formulas — plus a cheat sheet, practice questions, and exam strategy.

03 — Five Decisions

What shaped the product — and what each decision cost

3.1 — Chrome extension instead of paste-a-URL

The obvious design is a text box: paste a YouTube link, get a PDF. It broke immediately. YouTube aggressively blocks datacenter IPs — a server calling yt-dlp from Railway gets bot-challenged. Course platforms are auth-walled. Asking users to upload 200MB–2GB videos is bad UX and expensive storage.

The fix wasn't a feature — it was an architectural pivot: do the download-adjacent work in the user's browser. The extension uses the user's residential IP and authenticated session, scrapes captions client-side, relays only what the server needs.

Bought: bot-challenge disappears, auth-walled platforms become reachable, no upload step. Cost: real extension complexity, three sign-in paths, a cookie-rotation problem still handled clumsily. Otter, Tactiq, and Notta converged on the same architecture — which I take as validation the constraint, not cleverness, drove the answer.

3.2 — Concept-organized synthesis instead of time-sliced

The first synthesis pipeline sliced the transcript by time and summarized each slice. Output read like a transcript of the lecturer — "then he said X, then he wrote Y" — which is narration, not study material.

I changed the unit of synthesis from time to concept: extract canonical concept entities first, then build one card per concept by fusing every transcript window and visual extract where that concept appears.

Bought: notes organized by topic — everything about "F1 Score" in one place even if the lecturer returned to it three times. Cost: a harder extraction problem and the entire normalization layer described next.

3.3 — Deterministic normalization on top of stochastic output

This is the least visible decision and the one I'd most want a technical interviewer to see. The concept extractor is non-deterministic and wrong in predictable ways: it promotes passing mentions to full concepts, splits one concept across two names, treats sub-aspects as top-level, and orders by when things were said rather than how they build.

So the model's output isn't the answer — it's a candidate set. Thirteen deterministic passes run after it, in sequence, each shrinking or restructuring the set, each existing because something specific broke.

3.4 — Dynamic block schema instead of fixed sections

The first concept-card schema had fixed fields — definition, formula, example. It couldn't represent a concept with three formulas and no example without either truncating or emitting awkward empty sections.

V1 — Fixed Fields
Rigid. Always emits all fields.
definition: string
formula: string | null
example: string | null
pitfall: string | null
A 3-formula concept truncates. A definition-only concept emits empty sections. 24-page PDFs from 50-min lectures.
→
V2 — Dynamic Block Array
The model picks block order and shape.
{ type: "setup" | "intuition" | "why"
| "formula" | "deriv_step"
| "example" | "pitfall"
| "qa" | "diagram" | "note" }
Empty types don't render. 12–14 pages. Visual treatment (amber pitfalls, blue formulas) carries meaning without labels.

Schema design is product design. The V1→V2 shift changed what the model could express, not just how output was formatted.

3.5 — Source attribution as a first-class trust feature

Because the tool augments — filling in steps the lecturer skipped, adding intuition the lecturer assumed — every block carries a source tag: lecture or derived. The PDF renders subtle badges so a student knows which formula came off the slide and which the model added.

In a domain where mis-attribution is one of the dangerous-and-invisible failures, this is the central trust mechanism. The eval below reveals it's currently miscalibrated in the too-confident direction — covered honestly in Section 5.

05 — The Eval

What I designed versus what I shipped

What I designed. A four-tier evaluation framework: Tier 1 deterministic code checks on every job, Tier 2 LLM-as-judge for quality dimensions, Tier 3 reference-diff against golden fixtures, Tier 4 human verification. Specified per pipeline layer in the PRD.

What I actually shipped. A minimal, honest slice — not the whole thing, explicitly not CI-gated. Layer 1 deterministic checks live in-database as SQL views over per-job diagnostics. And a Layer 5 LLM-as-judge batch: 8-dimension rubric, run by an out-of-family judge (Claude, not Gemini, to avoid self-preference bias), structured JSON output with inline evidence fields for spot-checking. Run once, across six lectures. Two of six spot-checked against source material.

Six Lectures

ML theory (CS229, two lectures), MIT probability, IC-engines (NPTEL), biochemistry (IIT-KGP). Three with handwritten notes as reference, three with lecture screenshots. May 2026.

Three findings worth naming

The pipeline fails in the safer direction — but not the easier one

Zero invented formulas across 47. Zero fabricated Q&A. The error is almost entirely omission — missing concepts, missing formulas — not invention. Failing by omission is less catastrophic than failing by fabrication: an incomplete note is recoverable in a way a poisoned one isn't. But "less catastrophic" is not "easy to fix." Some omission is a recall problem addressable with window-boundary work; some isn't fixable downstream at all — a diagram the vision pass never captured, or a formula only stated aloud, can't be recovered by tuning synthesis. And all of it is invisible to the student. So: good news on the axis that matters most, paired with an honest limit.

Two card features had silently gone missing

Across all six lectures, in-card Q&A blocks and prerequisite content came back at zero — despite both being in the schema. This is a regression: Pass B had stopped emitting two feature classes, and nothing in production logging flagged it. The eval caught it on its first run. A 30-line deterministic check counting these blocks per job would have caught it in real time instead of weeks later in an offline batch. This is the single strongest argument for why eval-gated CI matters.

My own calibration fix overcorrected

Earlier in the build, source tags skewed too humble — the model labeled things the lecturer directly stated as derived. I fixed it with prompt calibration. The eval shows the fix pushed past center: three of six lectures now skew too confident (tagging AI-derived content as lecture), and none skew humble. Same bug, opposite direction. One round of calibration moved the bias rather than removing it.

Caveats

N = 6; pre-production batch evaluation, not statistically powered observability. Single judge with its own biases. Three of six references are screenshots — weaker ground truth than handwritten notes. Pedagogical accuracy is low-confidence by design — verifying whether a pitfall is real needs subject expertise the judge doesn't reliably have. Due to limited time, these evals were designed to provide directional signal rather than achieve abstract metric targets.

06 — What's Next

Five moves, in an order that matters

01
Scale the eval — and treat criteria drift as a feature
Six lectures show direction, not stable rates. Running thirty more naively misleads because as you grade more cases, your definition of "good" shifts — criteria drift. That's the eval working, not a flaw — each drift is a discovery that the rubric was underspecified. What you manage isn't the learning but the comparability: version the rubric, re-grade earlier lectures under each new version, validate the judge against human labels. Done right, a bigger batch sharpens the rubric and widens the failure catalogue. This comes first because it tells me whether the rest of the list is aimed correctly.
02
Wire Layer 5 deterministic checks into CI
The two silent regressions are the proof that regression detection beats incremental quality gains. Counting block types, source-tag distribution, and formula validity per job is cheap and would have caught finding 2 in real time — the batch finds new failure classes, CI stops known ones recurring.
03
Rebalance the source-tag prompt
Two-sided calibration examples, regression-tested against this batch to see whether the bias centers instead of flipping again.
04
Raise concept recall on dense lectures
IC-engines (44%) points at Pass A window boundaries missing the back half of long lectures — but this only recovers the fixable slice of omission; a diagram never captured or a formula only spoken aloud stays out of reach.
05
Validate and tune the LLM-as-judge with a domain expert
The judge's rubric reflects my own understanding of what "good" looks like — which is the wrong ground truth for pedagogical accuracy. A subject-matter expert reviewing a sample of graded outputs would surface where the rubric is underspecified, where the judge is systematically lenient or harsh, and whether dimensions like "pitfalls real" are being assessed correctly. Judge validation should precede rubric scaling.
Deliberately not next: diagram-block quality and pedagogical card ordering are open issues, but the eval didn't flag them as headline failures — the right signal they're not yet worth the spend.
07 — Reflection

Three things that hardened

Pattern 01
Deterministic scaffolding carries as much weight as the model call
The 13 normalization passes do as much to make the output trustworthy as the LLM does to generate it. AI-product portfolios rarely admit how much of the work lives in the non-model layers.
Pattern 02
Schema design is product design
The V1→V2 schema shift changed what the model could express, not just how output was formatted. The shape of the schema determines what the model is even able to say.
Pattern 03
Eval-first is harder to do than to claim
The framework you design and the framework you ship will always have a gap — and that gap is the most useful product signal you have. On its first run, the minimal eval did the job production logging lacked the discipline to do.
